Proxy Networks — расширение для Google Chrome на Manifest V3, которое хранит несколько прокси-профилей и позволяет быстро переключать HTTP, HTTPS и SOCKS5-прокси. Проект не требует сборки и не содержит сторонних runtime-зависимостей.

## Возможности

- несколько локальных прокси-профилей;
- теги и заметки для каждого профиля;
- перетаскивание профилей для изменения порядка;
- отдельные эндпоинты HTTP, HTTPS и SOCKS5;
- автоматическая маршрутизация по типу запроса или принудительный выбор одного протокола;
- режимы «весь трафик» и «только выбранные сайты»;
- профильный Kill Switch без прямого fallback при отказе прокси;
- прокси-авторизация по логину и паролю;
- вставка прокси из буфера в распространённых форматах;
- список адресов-исключений, а также встроенные исключения российских ресурсов и локальных сетей;
- компактные geosite-правила с локальным кэшем, ежедневным и ручным обновлением;
- список блокировки доменов и geosite-баз на уровне Chrome;
- опциональная блокировка трекеров, работающая независимо от состояния прокси;
- защита от утечек WebRTC;
- проверка каждого протокола с цветным индикатором и задержкой HTTPS-запроса;
- live-отображение текущего публичного IP и задержки контрольного HTTPS-запроса с обновлением каждые 3 секунды;
- страна, флаг, город и провайдер сохранённого или подключённого прокси с недельным локальным кэшем;
- диагностическая панель профиля с общей проверкой настроенных соединений;
- импорт и экспорт профилей в JSON, включая защищённый паролем экспорт учётных данных;
- динамическая иконка состояния: белая — выключено, зелёная — работает, красная — обнаружена ошибка.

## Требования

- Google Chrome 108 или новее;
- Node.js 18 или новее — только для запуска проверок проекта.

## Установка из исходников

1. Скачайте репозиторий или клонируйте его.
2. Откройте `chrome://extensions`.
3. Включите «Режим разработчика».
4. Нажмите «Загрузить распакованное расширение».
5. Выберите корневую папку проекта, содержащую `manifest.json`.

После изменения исходников нажимайте «Обновить» на карточке расширения.

## Использование

1. Откройте настройки расширения и создайте профиль.
2. Заполните хотя бы один прокси-эндпоинт или вставьте его из буфера, затем выберите режим маршрутизации и при необходимости укажите авторизацию и исключения.
3. Сохраните профиль.
4. В миниатюре выберите профиль и режим протокола.
5. Включите прокси переключателем.

В шапке миниатюры отображается версия из `manifest.json`. Блок статуса показывает флаг, страну, город и провайдера выбранного прокси, а также текущий публичный IP и задержку проверки. Если сохранён только один профиль или для него доступен один протокол, соответствующее поле остаётся видимым, но блокируется: значение уже выбрано и раскрывать список не требуется.

Режим «Авто» формирует правила Chrome для каждого заполненного эндпоинта: HTTP используется для HTTP-запросов, HTTPS — для HTTPS-запросов, а SOCKS5 — для остальных типов трафика или когда для типа запроса не настроен отдельный сервер. При ручном выборе один прокси применяется ко всем запросам.

### Режимы маршрутизации и Kill Switch

В профиле доступны два режима:

- «Весь трафик через прокси» — обычная конфигурация Chrome для всего поддерживаемого трафика с учётом исключений;
- «Только выбранные сайты через прокси» — расширение создаёт локальный PAC-скрипт, отправляет через прокси только домены из поля «Через прокси», а остальные адреса оставляет на прямом соединении.

Поле «Через прокси» поддерживает обычные домены, маски `*.example.com` и записи `geosite:имя`. Если один адрес одновременно присутствует в маршруте и исключениях, исключение имеет приоритет.

В выборочном режиме ручные поля «Исключения» и «Блокировать» скрыты и не применяются: ресурсы вне списка маршрутизации и так идут напрямую. Доступны только встроенные исключения российских ресурсов и локальных сетей — они имеют приоритет над маршрутом.

Kill Switch запрещает Chrome использовать `DIRECT` как запасной маршрут для трафика, который должен идти через прокси. Намеренно исключённые адреса и сайты вне выборочного маршрута продолжают открываться напрямую. Для проверки выборочного профиля служебный запрос к `api.ipify.org` также направляется через прокси.

### Оформление профилей

Для профиля можно добавить до 12 тегов и сохранить заметку. Теги отображаются в списке профилей и помогают различать серверы с похожими названиями. Профили по-прежнему можно переставлять перетаскиванием.

### Вставка прокси из буфера

Кнопка «Вставить из буфера» распознаёт одну строку прокси в следующих форматах:

```text
hostname:port
hostname:port:username:password
hostname:port@username:password
username:password:hostname:port
http://username:password@server:port
```

Поддерживаются URL и разделители `:`, `;`, `|`. Для специальных символов в URL-логине или пароле используйте URL-кодирование.

### Geosite-исключения

В список исключений можно добавлять записи из [v2fly/domain-list-community](https://github.com/v2fly/domain-list-community/tree/master/data) в компактном формате:

```text
geosite:google
geosite:cloudflare
```

Расширение загружает нужные базы из репозитория, обрабатывает их локально и передаёт Chrome только доменные шаблоны. Загруженные базы кэшируются в расширении и автоматически обновляются каждый день в 00:00. В правой панели настроек показаны размер и время обновления каждой используемой базы; там же доступно ручное обновление. При временной недоступности GitHub используется последний успешный кэш.

Поддерживаются правила `domain:`, `full:` и `include:`. Правила `keyword:` и `regexp:` пропускаются, поскольку Chrome `bypassList` не поддерживает их полноценно.

### Встроенные исключения

В настройках профиля есть две дополнительные галочки:

- «Исключить российские ресурсы» — добавляет домены из `rules/ru.json` в обход прокси;
- «Исключить локальные сети» — добавляет безопасные локальные адреса и сети из `rules/local.json` в обход прокси.

Эти правила применяются только при включённом профиле.

### Блокировка доменов

В поле «Блокировать» можно указать домены или geosite-базы:

```text
ads.example.com
geosite:magnit
```

При включённом прокси запросы к этим доменам блокируются самим Chrome через `declarativeNetRequest` и не получают ответ от сервера. При выключении прокси блокирующие правила удаляются.

## Проверка прокси

Кнопка «Проверить» временно применяет выбранный эндпоинт, запрашивает публичный IP через `api.ipify.org`, а затем восстанавливает предыдущую конфигурацию. Во время проверки сетевые запросы других вкладок могут на несколько секунд пройти через проверяемый прокси.

Зелёный индикатор означает успешный запрос, красный — ошибку, жёлтый — выполняющуюся проверку. После успешной проверки рядом с протоколом отображается задержка этого HTTPS-запроса. Результат автоматически скрывается через 10 секунд, а также сбрасывается после изменения адреса, порта или учётных данных.

### Защита WebRTC

Галочка в миниатюре включает политику Chrome `disable_non_proxied_udp`: она снижает риск раскрытия реального IP через WebRTC, запрещая непрофильный UDP-трафик. Защита может повлиять на WebRTC-звонки и P2P-соединения. Chrome не предоставляет расширению достоверный счётчик заблокированных WebRTC-запросов, поэтому в интерфейсе отображается только её текущий статус.

### Диагностика профиля

Правая панель настроек показывает состояние выбранного профиля, режим маршрутизации, число настроенных прокси и правил. Флаг, страну, город и провайдера каждого сохранённого прокси можно увидеть прямо в карточке профиля слева; если HTTP, HTTPS и SOCKS5-серверы различаются, там отображаются несколько строк. Те же сведения для выбранного протокола видны в миниатюре расширения. Для доменного хоста расширение сначала получает его IP через DNS-over-HTTPS, затем запрашивает геоданные; результат хранится локально до 7 дней.

Кнопка «Проверить соединения» последовательно проверяет каждый заполненный эндпоинт и выводит его публичный IP и задержку контрольного HTTPS-запроса. Как и одиночная проверка, эта операция временно меняет конфигурацию Chrome и затем восстанавливает её.

### Импорт и экспорт

Экспорт без паролей создаёт обычный читаемый JSON. При экспорте с паролями расширение обязательно запрашивает отдельный пароль файла и шифрует всё содержимое с помощью AES-256-GCM; ключ выводится из пароля через PBKDF2-SHA-256. Пароль файла нигде не сохраняется и потребуется при импорте.

Старые незашифрованные JSON-файлы импорта продолжают поддерживаться. Потерянный пароль защищённого экспорта восстановить невозможно.

## Разрешения

| Разрешение | Назначение |
| --- | --- |
| `proxy` | Применение конфигурации через `chrome.proxy.settings` |
| `storage` | Локальное хранение профилей и состояния |
| `alarms` | Ежедневное обновление кэша geosite-баз |
| `privacy` | Управление политикой защиты WebRTC |
| `clipboardRead` | Вставка прокси из буфера обмена по действию пользователя |
| `declarativeNetRequestWithHostAccess` | Профильная блокировка доменов и опциональная блокировка трекеров |
| `webRequest` | Получение событий прокси-аутентификации |
| `webRequestAuthProvider` | Передача логина и пароля в proxy-auth challenge |
| `<all_urls>` | Работа прокси и авторизации для любых открываемых адресов |

## Данные и безопасность

Все профили хранятся в `chrome.storage.local`. Пароль шифруется AES-GCM, однако ключ находится в том же локальном хранилище. Это защита от случайного просмотра, а не от пользователя или программы с полным доступом к профилю Chrome.

Доступ к `chrome.storage.local` ограничивается доверенными контекстами самого расширения. Geosite-кэш автоматически обрезается до безопасного размера, чтобы не вытеснять профили и учётные данные из квоты Chrome.

Экспорт без паролей используется по умолчанию. Экспорт с паролями защищается отдельным пользовательским паролем, но стойкость такого файла зависит от сложности выбранного пароля. Не публикуйте экспортированные профили и не добавляйте реальные учётные данные в Git.

## Ограничения

- прокси применяется ко всему обычному профилю Chrome, а не к отдельной вкладке;
- поддерживаются HTTP, HTTPS и SOCKS5; выборочная маршрутизация и Kill Switch используют локально сгенерированный PAC-скрипт, а пользовательские PAC-URL не поддерживаются;
- SOCKS5 работает без авторизации: Chrome не поддерживает логин и пароль для SOCKS5-прокси;
- расширение не управляет отдельной конфигурацией прокси для инкогнито;
- доступность прокси определяется HTTPS-запросом к `api.ipify.org`, поэтому сбой этого сервиса или блокировка домена также даст отрицательный результат;
- показываемый «пинг» — полная задержка этого HTTPS-запроса через текущий маршрут, а не ICMP-пинг непосредственно до proxy-сервера;
- Chrome или другое расширение с разрешением `proxy` может переопределить текущую конфигурацию.

## Структура проекта

```text
.
├── manifest.json         # Manifest V3
├── background.js         # service worker, авторизация и проверки
├── config.js             # построение правил chrome.proxy
├── geo.js                # определение и кэширование локации прокси
├── geosite.js            # загрузка, разбор и кэширование geosite-баз
├── blocker.js            # динамические правила блокировки Chrome
├── storage.js            # хранение профилей и шифрование паролей
├── utils.js              # общие Promise-обёртки и нормализация данных
├── rules/                # локальные сети, российские ресурсы и трекеры
├── popup.*               # миниатюра расширения
├── options.*             # редактор профилей
└── images/               # брендовые и статусные PNG
```

## Диагностика

Если Chrome сообщает `ERR_TUNNEL_CONNECTION_FAILED`, проверьте адрес, порт и авторизацию нужного протокола. Низкоуровневую информацию о прокси можно посмотреть на `chrome://net-internals/#proxy`, а ошибки service worker — на карточке расширения в `chrome://extensions`.
